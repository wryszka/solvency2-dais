# Solvency II demo — Reality + Redeployability audit

**Top 5 redeployability blockers.** (1) `src/app/app.yaml` hardcodes `CATALOG_NAME=lr_serverless_aws_us_catalog`, `SCHEMA_NAME=solvency2demo_agentic`, `WAREHOUSE_ID=ab79eced8207d29b`, `DASHBOARD_ID=01f128f6dc951605aca6caa3ec26edb3`, `GENIE_SPACE_ID=01f128fe84201b87ae7c04614b375d6c` — these are not driven by `databricks.yml` variables and a fresh deploy will read someone else's workspace. (2) Multiple scripts (`scripts/seed_phase5.py:24-26`, `scripts/seed_governance.py:27-29`, `scripts/preflight_check.sh:14-18`, `scripts/create_dashboard.py:9-11`, `scripts/create_dashboard_v2.py:24-28`, `deploy_demo.sh:27`, `remove_demo.sh:17-19`) hardcode dev catalog/schema/warehouse defaults that do not match `databricks.yml` defaults and have no consistent override mechanism. (3) The Genie space and Lakeview dashboard are created by imperative Python scripts (`scripts/create_dashboard*.py`, `deploy_demo.sh:248-286`) rather than declared as bundle resources — their UUIDs are baked back into `app.yaml` manually. (4) The Standard Formula MLflow model and the reserving models live outside the bundle and are registered by ad-hoc job submissions in `deploy_demo.sh:188-211` plus `governance_setup_job.yml`; a fresh clone has no way to know whether v1/v2 + Champion/Challenger aliases were created. (5) `scripts/preflight_check.sh` has 38 SQL/HTTP probes but no verification of: model registration with correct aliases, app health beyond reachability, agent end-to-end behaviour, regulatory_exports volume existence, or `Reset Demo` working.

### B2 — Out-of-bundle workspace state

These objects are required for the demo but are **not declared in `databricks.yml` or `resources/*.yml`**, so `databricks bundle deploy` on a fresh workspace will not recreate them:

1. **SQL Warehouse** — used by every API call. Discovered by hardcoded ID in `app.yaml:8` and scripts. **Bundle home:** add to `resources/warehouse.yml` or instruct user to set `WAREHOUSE_ID` env override.
2. **Lakeview dashboard** — created imperatively by `scripts/create_dashboard.py` / `scripts/create_dashboard_v2.py`; ID then pasted manually into `app.yaml:10`. **Bundle home:** `resources/dashboard.yml` declaring a `dashboard:` resource.
3. **Genie space** — created imperatively in `deploy_demo.sh:248-286`; ID pasted into `app.yaml:12`. **Bundle home:** Genie spaces are not yet a first-class bundle resource — best path is a one-shot job task that creates and writes back the ID, or move to a config table the app reads.
4. **Databricks App** (`solvency2-qrt-ai` or `solvency2-qrt-ai-dev`) — created via `databricks apps create` in `deploy_demo.sh:294`. **Bundle home:** `resources/app.yml` declaring an `app:` resource.
5. **MLflow registered model** `standard_formula` (versions 1 + 2, aliases Champion/Challenger and production/candidate/archive) — registered by `src/03_QRT_S2501_SCR/register_standard_formula_model.py` invoked via the deploy script (`deploy_demo.sh:188-211`), not via the bundle. **Bundle home:** the existing `resources/qrt_s2501_pipeline.yml` could add a `register_model` task; alias state still needs a runtime job.
6. **Reserving models** (`reserving_pnc`, `reserving_life`) — registered by `src/02_Reserving_Model/register_reserving_models.py`, called from `governance_setup_job.yml`. The job IS in the bundle but it has to be `bundle run` separately; not part of `bundle deploy`. **Bundle home:** chain `governance_setup` to the QRT pipelines via dependencies.
7. **`6_gov_*` governance tables** — populated by `src/01_Bootstrap_Governance/bootstrap_governance.py` (in bundle via `governance_setup_job.yml`) AND by `scripts/seed_governance.py` (out-of-bundle laptop path). The two paths can drift.
8. **`6_demo_*` Phase-5 tables** — created and seeded by `scripts/seed_phase5.py` only (laptop-run). **Bundle home:** wrap in a notebook + add a `seed_phase5_demo` job task to `governance_setup_job.yml` so a fresh deploy populates them.
9. **`gold_submissions_archive`, `gold_orsa_draft`** — DDL is in `seed_phase5.py:130-143`; the app's `_rebase_demo_state` only DELETEs from these (`demo.py:851`), it doesn't CREATE. Without seed_phase5 being run, Reset Demo will throw. **Bundle home:** same as above.
10. **`6_ai_demo_cache`** — created lazily by the app at first cache write (`cache.py:35-41`); pre-baked entries come from `scripts/bake_cache.sh`. Not declared anywhere.
11. **Regulatory exports Volume** (`/Volumes/{catalog}/{schema}/regulatory_exports`) — referenced in `approvals.py:334` and `358` but **never created**. First approve will warn `volume_path (upload pending)` (line 343). **Bundle home:** `resources/volume.yml` declaring the volume.

### B5 — Idempotency gaps

1. `deploy_demo.sh` re-creates all DLT pipelines and jobs via `databricks bundle deploy` (idempotent), but the **Genie space create** (`deploy_demo.sh:278`) does a POST without a "look up existing first" step — re-runs will create *another* space and the new ID is then dropped on the floor (it's not written back to `app.yaml`).
2. **Standard Formula model registration** runs every deploy and accumulates new MLflow versions. `register_standard_formula_model.py` registers v1 + v2 each call → after N deploys you have versions 1–2N (the ones with proper aliases get overwritten but old artefacts stick around).
3. **Reserving model registration** has the same accumulation problem — `register_reserving_models.py` (referenced from `governance_setup_job.yml:25`) is run every time the job runs.
4. `_ensure_orsa_tables` (`orsa.py:117`), `cache.ensure_cache_table`, `approvals.ensure_approvals_table`, `audit.ensure_agent_audit_table`, `model_governance._ensure_tables` all use `CREATE TABLE IF NOT EXISTS` — safe, but **column drift is silent**: if the schema needs to change, the existing table just keeps the old columns and the INSERT errors out at runtime.
5. `seed_phase5.py` calls `DELETE FROM` then re-inserts (`scripts/seed_phase5.py:67-77`) — idempotent but loses any user-created rows on `6_demo_*` tables (e.g. what-if runs).
6. `_rebase_demo_state` (`demo.py:636-888`) `DELETE`s+re-inserts feeds, solvency series, ORSA history every time. If the function fails halfway through (e.g. during the storm UPDATE at `demo.py:822`), tables are left in a partial state — no transaction wrapping.
7. `bootstrap_archive.py` regenerates Q1-Q3 history; if it runs twice with different reporting_year arg, results compound (no `WHERE NOT EXISTS` guard observed in the parts of the file inspected; it seeds via overwrite paths but I did not exhaustively verify all targets).
8. The dev_v2 `set_alias` calls in `demo.py:117-122` and `demo.py:961-963` use literal `version_num=1` / `version_num=2`. If the model has been re-registered enough times that v1/v2 are no longer the right versions, the alias flip points at the wrong version.
9. `deploy_demo.sh:181` deletes `.databricks/bundle/dev/sync-snapshots` before `bundle deploy` — fine for cleanliness but means **no incremental deploy**; full re-sync every time.
10. The app middleware writes to `5_mon_agent_audit` on every request (`app.py:138-159`); failures are swallowed (`app.py:160-161`). If the table schema drifts, the app will silently lose audit rows — no surfacing.
11. `seed_phase5.py:155` uses `NOW = datetime(2025, 12, 8, ...)` as the anchor for the seeded series. The in-app `_rebase_demo_state` overrides this with real "today", but if `seed_phase5.py` is the only step that ran (no Reset clicked), the demo data is anchored in late 2025.
12. `Reset Demo` cleans overlays via `WHERE created_at > CAST('2026-01-15 00:00:00' AS TIMESTAMP)` (`demo.py:982`) — this hardcoded cutoff means after that date drifts further into the past, real overlays may be deleted along with demo-created ones.

### B6 — Verification gaps

1. **No model-registration check.** Preflight does not assert that `{catalog}.{schema}.standard_formula` exists with `Champion` AND `Challenger` aliases attached — both are required for the model-governance comparison view.
2. **No reserving-model check.** `reserving_pnc` and `reserving_life` registration is not asserted.
3. **No app-health-beyond-reachability check.** `probe_endpoint` accepts `200`, `302`, or `401` (`preflight_check.sh:88-93`) — a 401 means "the proxy is up" but not that the app process is. There is no probe that actually returns app data (e.g. fetch `/api/landing/status` and verify a tile is populated).
4. **No agent end-to-end test.** Preflight does not exercise the FM endpoint path. If `databricks-claude-sonnet-4` is not enabled on the target workspace, the cat agent / ORSA narrative / second-opinion / workbench / senior reserving / recon-investigate / supervisor will all 500 at runtime — the preflight passes.
5. **No Genie verification.** `GENIE_SPACE_ID` could be wrong/orphaned after re-deploy; no probe runs an actual Genie question.
6. **No dashboard verification.** `DASHBOARD_ID` could be stale; no probe loads the embed URL.
7. **No volume-existence check.** Approvals' `regulatory_exports` and Igloo's `4_eng_stochastic_exchange` volumes — first approval/run will silently degrade (warning-logged) if missing.
8. **No `Reset Demo` smoke test.** The Reset path is critical (it sets the time anchor) but is not exercised by preflight.
9. **Row-count thresholds rather than presence.** Several preflight probes use `LIMIT 1` (e.g. `5_mon_pipeline_sla_status` line 112) — they pass even if there is only 1 stale row from months ago. No "row produced this week" check.
10. **Phase-5 tables (`6_demo_*`) not in preflight.** Without `seed_phase5.py` having been run, `Today`'s solvency tile, late-feed callout, SF Challenger panel, and ORSA continuous tile all break; preflight does not probe `6_demo_data_feeds`, `6_demo_solvency_daily`, `6_demo_sf_challenger`, etc.
11. **No grant verification.** Preflight does not run `SHOW GRANTS ON SCHEMA ... TO {APP_SP}` to confirm the app SP has the access it needs.
12. **No Lakeview dashboard / Genie space matching `app.yaml`.** Preflight does not assert that `DASHBOARD_ID` / `GENIE_SPACE_ID` from app.yaml resolve to existing assets in the target workspace.
13. **No Phase-5 seed check.** `gold_submissions_archive` and `gold_orsa_draft` are required by the Submissions page and ORSA Draft; preflight does not check they have rows.
14. **No FM endpoint discovery test.** A `databricks serving-endpoints get databricks-claude-sonnet-4` would catch the most common "fresh workspace" failure cause in 1 call.
